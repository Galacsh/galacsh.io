배경지식

인터넷상의 텍스트 기반 메일 메시지의 형식을 정의하는 RFC 822 는 US-ASCII 텍스트만을 지원했습니다. 그러다보니 이외의 Character Set 이나 텍스트가 아닌 각종 미디어 데이터는 표현할 수 없었고 이를 재정의하기 위해 MIME(Multipurpose Internet Mail Extensions)이 나타났습니다.

  • Multipurpose Internet Mail Extensions (MIME)
    • RFC 2045: Part One: Format of Internet Message Bodies
    • RFC 2046: Part Two: Media Types
    • RFC 2047: Part Three: Message Header Extensions for Non-ASCII Text
    • RFC 2048: Part Four: Registration Procedures
    • RFC 2049: Part Five: Conformance Criteria and Examples

위 문서들 중 Part Two에서 정의된 Media Types를 이 글에서 간략히 다룰 예정입니다. 참고로, MIME Types는 이 Media Types의 예전 명칭이라고 합니다.

... Media Types (formerly known as MIME types) ... 출처: Media Types (iana.org)

Media Types

간단히 이해하자면, 데이터를 주고 받을 때 포함되는 미디어가 텍스트인지 이미지인지 등을 명시하는 표준 표기법 + 미디어 종류 목록 정도로 생각하면 됩니다.

전체 미디어 종류는 IANA에서 확인할 수 있습니다.

RFC2046 specifies that Media Types (formerly known as MIME types) and Media Subtypes will be assigned and listed by the IANA.
출처: Media Types (iana.org)

구조

Media Type은 다음과 같은 구조로 이루어져 있습니다.

[Top-level media type]/[Subtype]; [Parameter1 key]=[Parameter1 value]; ...; [ParameterN key]=[ParameterN value]
  • 예시
    • image/jpeg
    • audio/mp4
    • text/plain; charset=UTF-8

Top-level Media Types

작성 시점 기준으로는 다음과 같은 상위 미디어 종류들이 지정되어 있습니다.

multipart/form-data

주로 파일 업로드 기능 구현 시 마주치게 되는 Media Type입니다. 파일 업로드 기능을 구현할 때는 보통 다음과 같이 <form>...</form> 을 구성하거나,

<form action="/somewhere" method="post">
  <label>Text: <input name="myText" value="Test" /></label>
  <label>
    Upload file: <input type="file" name="file" value="test.txt" />
  </label>
  <button type="submit">전송</button>
</form>

직접 FormData 객체를 생성해 전송합니다.

var input = document.querySelector('input[type="file"]')

var formData = new FormData()
formData.append('myText', "Test")
formData.append('myFile', input.files[0])

fetch('/somewhere', {
  method: 'POST',
  body: formData
})

이렇게 전송하게 되면, 다음과 같은 형식으로 메시지를 전송하게 됩니다.

POST / HTTP/1.1
...생략
Content-Type: multipart/form-data; boundary=boundarytext

Content-Length: 

--boundarytext
Content-Disposition: form-data; name="myText"

Test
--boundarytext
Content-Disposition: form-data; name="myFile"; filename="test.txt"
Content-Type: text/plain

Simple file.
--boundarytext--

값들 간의 경계 영역을 표시하기 위한 boundary 값과, 이를 이용해 각 데이터가 구분되어 보내집니다.

위와 같이 fetch API를 이용해 form data를 전송할 때는 주의할 점이 있는데, 직접 contentType을 지정하는 경우입니다. boundary 파라미터가 필수이다 보니, 직접 contentType을 지정하면서 boundary 파라미터를 지정하지 않으면 Boundary Not Found 에러를 마주하게 됩니다.

Boundary Not Found 에러를 마주했다면 위에서 예시로 들었던 사용례와 같이 아예 contentType을 지정하지 않는 방법으로 자동으로 지정이 되게끔 처리하거나, form data 내에 존재하지 않는 유니크한 문자열을 boundary 파라미터로 직접 지정해줘야 합니다.

Boundary 파라미터 및 boundary delimeter line에 대한 규칙은 아래 링크를 참고하세요.
5.1. Multipart Media Type

사용처

MIME에서는 Content-Type 헤더 필드를 이용해 Media Type을 지정하며, HTTP 프로토콜 또한 마찬가지이나 Accept 헤더 필드를 이용하여 응답 값에 포함될 희망 미디어 범위를 지정하기도 합니다.

예시:

Content-Type: text/plain; charset=UTF-8
Accept: audio/*; q=0.2, audio/basic

Accept 의 경우 명확한 미디어 타입이 아닌 미디어 범위(range)를 지정합니다.
